# Dharmiq v0.3 — Design System Implementation Plan

**Status:** Implemented (2026-06-21)  
**Purpose:** Step-by-step playbook to implement the finalized Ashoka design system (v0.3) on the existing React frontend, grounded in the current codebase.  
**Design authority (in order):**

1. [`docs/design/dharmiq-design-demo.html`](../design/dharmiq-design-demo.html) — pixel-accurate visual + interaction spec (**wins on conflict**)
2. [`docs/design/README.md`](../design/README.md) — written principles, values, component specs, stack mapping
3. [`docs/design/tokens.json`](../design/tokens.json) — portable tokens (sync to demo; do not ship archival/gazette)

**Baseline:** v0.2 frontend — functional async chat, SSE progress, streaming answers, uploads, session attachments, full-page document viewer (`/docs/:id`), no Settings page, no i18n, no theme toggle, placeholder shadcn/OKLCH theme.

> **Do not copy the demo HTML into the repo.** Re-implement look and behavior in React 19 + Tailwind v4 + shadcn + assistant-ui, driven by CSS variables derived from the demo.

---

## How to use this document (for implementing agents)

1. **Read binding decisions first** — §3 (resolved discrepancies), §4 (locked architecture), and §5 (product decisions). v0.3 is **design-only** — no new backend APIs except `tokens.json` sync (D12). Backend work is listed in §13 for v0.4.
2. **Do not scope-creep into v0.4** — if a task needs a new API, ship the visual/UI with the documented stub and move the API to §13.
3. **One phase at a time.** Do not start phase N+1 until that phase’s smoke tests pass.
4. **Match existing conventions:** `@/` imports, shadcn primitives in `frontend/src/components/ui/`, assistant-ui primitives in `frontend/src/components/assistant-ui/`, Tailwind v4 `@theme` in `frontend/src/index.css`.
5. **Every phase ends with:** code + manual visual check against the demo (same viewport: desktop + mobile toggles in demo top bar) + `npm run build` + `npm run lint`.
6. **Do not commit** unless the user asks.

### Global smoke gate (after every frontend phase)

```bash
cd frontend
npm run lint
npm run build
```

Open `docs/design/dharmiq-design-demo.html` side-by-side with `http://localhost:5173` and compare the phase’s surfaces in **Light**, **Dark**, and **Mobile** modes.

| Dimension | Value |
|-----------|-------|

## 4. Locked architecture decisions

### 4.1 Theme application

- Add `frontend/src/providers/ThemeProvider.tsx`:
  - Reads/writes `localStorage` key `dharmiq_theme`: `"light"` | `"dark"`.
  - Applies/removes `dark` class on `document.documentElement`.
  - Default: `"light"`.
  - **Do not** use `prefers-color-scheme` for automatic dark (user control via Settings + top-level toggle).
- Wrap in `main.tsx`: `ThemeProvider` → `AuthProvider` → app.

### 4.2 App shell component tree

Create `frontend/src/components/layout/AppShell.tsx`:

```text
AppShell
├── AuroraBackground (fixed, z-0, pointer-events-none)
├── grid: sidebar | main-column | DocumentPanel
│   ├── AppSidebar (desktop) — 250px / 70px collapsed
│   │   └── nav + ThreadList (v0.2 pattern, §5.2) + user footer
│   ├── MainColumn
│   │   ├── TopNav (breadcrumb, grounded chip, progress toggle)
│   │   ├── Outlet (Chat | Documents | Settings)
│   │   └── MobileTabBar (≤ md breakpoint)
│   └── DocumentPanel (desktop: resizable; mobile: overlay)
└── MobileAppBar (≤ md)
```

- `ChatPage` becomes thin: `{ session attachments provider, thread }` inside shell.
- New `DocumentsPage.tsx`, `SettingsPage.tsx`.
- `DocumentViewerPage` refactored: export presentational `DocumentPanel` used by shell; route `/docs/:id` renders shell with panel open (mobile full-screen).

### 4.3 Document panel open state

- URL-driven where possible: `/docs/:documentId?source_type=corpus&chunk=:chunkId&quote=:encoded` opens panel.
- Clicking in-chat citation: `navigate` with query params **or** internal state + `history.pushState` — prefer **React Router** navigation to keep deep links shareable.
- Closing panel returns to current page route without `/docs/...`.

### 4.4 Citation click behavior

- From chat: open side panel (desktop) or overlay (mobile); do **not** unmount thread.
- Direct visit to `/docs/:id`: acceptable full-panel experience (mobile-first).

### 4.5 Progress view tiers

| Tier | Storage | UI location | Backend |
|------|---------|-------------|---------|
| Concise | `dharmiq_progress_view=concise` | Default | SSE `filter_event_payload_for_view` strips agent/counts |
| Detailed | `detailed` | Top nav + Settings | Shows agent, chunk_count, preview |
| Debug | N/A (not persisted for regular users) | **Keep existing `DebugPanel`** below top nav (superuser only) | SSE debug fields; existing `DebugPanel` data |

Regular users never see debug payloads (already enforced server-side per R4-9).

### 4.6 Font roles

| Role | CSS variable | Applied to |
|------|--------------|------------|
| UI / body | `--font-sans` → Inter | body, buttons, nav, composer |
| Display | `--font-display` → Fraunces | h1–h6 in answers, page titles, wordmark |
| Mono | `--font-mono` → Geist Mono | citation refs, counts, doc viewer lines, progress `.det` |
| Hindi override | `[lang="hi"]` or `html[lang="hi"]` | Lead stack with Noto Sans Devanagari; §8.3 |

## 6. Behavioral specs (demo → code)

### 6.1 Progress step aggregation (concise view)

Map backend `step_id` → display bucket:

| Display order | Display label (EN) | Backend step_ids absorbed | Detailed mono hint (examples) |
|---------------|-------------------|---------------------------|-------------------------------|
| 1 | Understanding your question | `input_guard`, `clarifier` | `clarifier · …` |
| 2 | Searching the law | `query_rewriter`, `retrieve` (running phase) | `corpus+uploads · N chunks` |
| 3 | Ranking the best sources | `retrieve` (completed; use chunk_count) | `rerank · top N` |
| 4 | Checking the answer | `answerer`, `citation_enricher`, `validator`, `refusal` | `validator · claims n/m` |
| 5 | Writing the answer | `finalizer` | `finalizer` |

**Rules:**

- A display step **running** if any absorbed backend step is `running`.
- **done** if all absorbed steps `completed` (failed → show failed state on active bucket).
- **Detailed** row shows `step.agent`, `step.chunkCount`, etc. from the **most recent** completed backend step in that bucket.
- **Refusal** path: when `refusal` completes, bucket 4 label may read “Checking the answer” still — final answer is refusal card, not stream.

Implement in `frontend/src/lib/progressDisplay.ts` (new) consumed by `MessageProgress.tsx`.

### 6.2 Motion (respect `prefers-reduced-motion: reduce`)

| Token | Value |
|-------|-------|
| `--ease-default` | `cubic-bezier(0.22, 1, 0.36, 1)` |
| `--ease-enter` | `cubic-bezier(0.16, 1, 0.3, 1)` |
| `--ease-spring` | `cubic-bezier(0.34, 1.56, 0.64, 1)` (chips/pop only) |
| `--duration-instant` | 100ms |
| `--duration-fast` | 150ms |
| `--duration-normal` | 220ms |
| `--duration-slow` | 380ms |

Animations: message fade+7px rise (stagger 90ms), step pulse, chip pop-in, aurora 22s, doc-line highlight pulse, auth card spring pop — all disabled/reduced to opacity-only when `prefers-reduced-motion`.

### 6.3 Layout breakpoints

- **Mobile shell:** `max-width: 420px` behavior in demo ≈ Tailwind `md:` breakpoint (default `768px`) — tune to `max-md:` for app bar + bottom tabs. Verify at 390px and 4K.
- **Doc panel overlay:** `max-md:` full-screen slide from right; no resizer.

### 6.4 Structured answer sections

Headings expected in markdown (from answerer prompt): Summary, What the law says, What your document says, How this applies, Practical next steps, Assumptions, Disclaimer.

Style `h5`/`h2` in markdown with Fraunces. Disclaimer: append if not in markdown via `final_warning` from stream done event — render demo `.disc` block.

| Risk | Mitigation |

| assistant-ui default styles fight custom bubbles | Override via `aui-*` classes; avoid forking library |
| PDF-only viewer cannot show mono line highlights | v0.4 chunk API (§13); v0.3 iframe + optional banner |

| Privacy settings without backend | Q8: omitted from Settings in v0.3 |
